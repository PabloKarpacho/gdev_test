Как устроен Telegram ChatGPT Bot
01

Что происходит, когда вы пишете боту

Этот репозиторий собирает компактного Telegram-бота, который умеет отвечать на текст, фото и голос. Главная архитектурная идея простая: каждое звено делает только одну работу, а затем передает эстафету дальше.

Начинаем с продукта, а не с Python

Представьте почтовый терминал: вы кладете конверт в окно, а дальше его подхватывает целая цепочка сотрудников. Здесь такой же принцип. Пользователь видит один чат, но внутри работают Telegram API, handler, сервис и AI-клиент.

1
Текст

Самый короткий маршрут: сообщение пришло, сервис проверил пустой ли текст, затем вопрос ушел в модель.

2
Фото

Сначала бот скачивает самую крупную версию изображения, проверяет размер и только потом отдает его AI.

3
Голос

Аудио превращается в текст через транскрипцию, и уже этот текст идет в обычный диалоговый ответ.

Код, где бот собирается в одно целое

Источник: bot/__main__.py, строки 23-35. Это entrypoint всей системы. Как режиссер перед съемкой, он собирает актеров на площадке и раздает роли.

CODE 
settings = load_settings()
setup_logging(level=settings.log_level)
logger.info("Starting bot application")

bot = get_bot(token=settings.bot_token)
dispatcher = get_dispatcher()
ai_client = build_openai_chat_client(
    api_key=settings.openai_api_key,
    model_name=settings.openai_chat_model,
    transcription_model=settings.openai_transcription_model,
    system_prompt=settings.system_prompt,
)
chat_service = ChatService(ai_client=ai_client)
PLAIN ENGLISH

Сначала программа читает настройки из окружения и готовит логирование.

Потом создается клиент Telegram, то есть объект, который умеет говорить с сервером Telegram.

Следом поднимается dispatcher: он будет решать, какой обработчик должен сработать на новое событие.

Дальше собирается AI-клиент с нужной моделью, ключом и системным промптом.

В самом конце создается ChatService, который становится центральным координатором между Telegram и OpenAI.

Источник: bot/handlers/__init__.py, строки 26-29. Это хороший пример разделения ответственности: файл не обрабатывает сообщения сам, а только регистрирует нужные маршруты.

CODE

dispatcher.include_router(build_start_router())
dispatcher.include_router(build_chat_router(chat_service=chat_service))

PLAIN ENGLISH

Сначала код пишет в лог, что начинает подключать маршруты.

Потом в dispatcher добавляется отдельный роутер для команды /start.

Следом подключается чатовый роутер, которому сразу передают готовый ChatService.

В конце система фиксирует, что маршрутизация собрана и бот готов принимать события.

Как общаются три режима: text, photo, voice

Пользователь видит одну кнопку отправки, но архитектурно внутри живут три разных сценария. Это хороший пример того, как один продуктовый интерфейс прячет несколько технологических маршрутов.

Три маршрута, три разных уровня сложности

✍️

Самый простой путь: обрезать пробелы, проверить пустоту и вызвать модель.

Нужна работа с бинарным вложением: скачать, проверить размер, собрать multimodal запрос.

Код, где сервис выбирает поведение для фото

Источник: bot/services/chat.py, строки 80-85. Это хороший пример того, зачем нужен service layer: он принимает уже скачанный файл и решает, с каким prompt отправлять его дальше.

CODE 
prompt = (caption or "").strip() or DEFAULT_PHOTO_PROMPT
return await self._ai_client.respond_to_photo(
    prompt=prompt,
    photo_bytes=attachment.content,
    mime_type=attachment.mime_type,
)
PLAIN ENGLISH

Если подписи к фото нет, сервис сам подставляет безопасный стандартный prompt.

Дальше он не думает про Telegram и не возится с файлами заново, а просто зовет AI-клиент.

В AI уходят три вещи: текстовая инструкция, сами байты картинки и MIME type.

Благодаря этому сервис отвечает за смысл запроса, а не за сетевые детали.

Что меняется между режимами на практике

T
Текстовый режим

Минимум инфраструктуры. Идеален как базовый канал и как fallback-модель мышления для всей системы.

P
Фото-режим

Добавляется binary attachment и проверка размера, потому что цена ошибки тут выше.

V
Голосовой режим

Появляется промежуточное представление: сначала аудио, потом текст, и только затем обычный AI-ответ.

Какие настройки действительно двигают систему

Хорошая конфигурация похожа на паспортный контроль: наружу выпускаются только те данные, без которых система не может жить. Все остальное либо имеет дефолт, либо игнорируется.

BOT_TOKEN Позволяет приложению действовать от имени Telegram-бота
OPENAI_API_KEY Открывает доступ к внешнему AI-провайдеру
OPENAI_CHAT_MODEL Выбирает модель для ответов на текст и фото
OPENAI_TRANSCRIPTION_MODEL Выбирает модель для расшифровки голосовых
SYSTEM_PROMPT Формирует характер и стиль ответа ассистента

Код, где голос выходит за пределы приложения

Источник: bot/ai/client.py, строки 173-182. Здесь видна граница с внешним SDK: локальные байты превращаются в запрос к OpenAI, а затем обратно в обычный текст.

CODE 
buffer = BytesIO(audio_bytes)
buffer.name = filename
response = await self._transcription_client.audio.transcriptions.create(
    model=self._transcription_model,
    file=buffer,
    response_format="text",
)
if isinstance(response, str):
    return response.strip()
return getattr(response, "text", "").strip()
PLAIN ENGLISH

Сырые байты голоса оборачиваются во временный файловый объект, потому что внешнее API ждет именно файл.

К объекту добавляется имя файла, чтобы сервис понимал, с чем работает.

Дальше вызывается удаленная транскрипция: модель, файл и формат ответа отправляются наружу.

Когда ответ приходит обратно, код аккуратно нормализует его в обычную строку без лишних пробелов.

Код, который не бросает ресурсы при сбое

Источник: bot/run.py, строки 47-68. Это архитектурный момент важнее, чем кажется: хороший процесс думает не только о запуске, но и о том, как корректно завершиться.

CODE 
try:
    logger.info("Deleting webhook before polling")
    loop.run_until_complete(bot.delete_webhook())
    logger.info("Setting up bot and dispatcher")
    loop.run_until_complete(
        setup_bot(
            bot=bot,
            dispatcher=dp,
            chat_service=chat_service,
        )
    )
finally:
    logger.info("Closing bot session")
    loop.run_until_complete(close_bot_session(bot))
PLAIN ENGLISH

Система пробует запустить polling по обычному плану: убрать webhook и подготовить бота.

Даже если на этапе setup случится авария, код не выходит молча и не забывает про сетевую сессию.

Блок finally срабатывает почти как обязательная уборка в конце смены: сначала закрыть соединение, потом отпускать процесс.
