Какие настройки действительно двигают систему

Хорошая конфигурация похожа на паспортный контроль: наружу выпускаются только те данные, без которых система не может жить. Все остальное либо имеет дефолт, либо игнорируется.

BOT_TOKEN Позволяет приложению действовать от имени Telegram-бота
OPENAI_API_KEY Открывает доступ к внешнему AI-провайдеру
OPENAI_CHAT_MODEL Выбирает модель для ответов на текст и фото
OPENAI_TRANSCRIPTION_MODEL Выбирает модель для расшифровки голосовых
SYSTEM_PROMPT Формирует характер и стиль ответа ассистента

Практический вывод

Если AI-агент предлагает вписать ключ прямо в Python-файл, это уже архитектурный запах. В этом проекте секреты намеренно живут только в окружении.

Код, где голос выходит за пределы приложения

Источник: bot/ai/client.py, строки 173-182. Здесь видна граница с внешним SDK: локальные байты превращаются в запрос к OpenAI, а затем обратно в обычный текст.

CODE 
buffer = BytesIO(audio_bytes)
buffer.name = filename
response = await self._transcription_client.audio.transcriptions.create(
    model=self._transcription_model,
    file=buffer,
    response_format="text",
)
if isinstance(response, str):
    return response.strip()
return getattr(response, "text", "").strip()
PLAIN ENGLISH

Сырые байты голоса оборачиваются во временный файловый объект, потому что внешнее API ждет именно файл.

К объекту добавляется имя файла, чтобы сервис понимал, с чем работает.

Дальше вызывается удаленная транскрипция: модель, файл и формат ответа отправляются наружу.

Когда ответ приходит обратно, код аккуратно нормализует его в обычную строку без лишних пробелов.
